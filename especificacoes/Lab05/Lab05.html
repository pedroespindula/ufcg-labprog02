Introduo

Neste laboratrio comearemos a trabalhar atravs de uma especificao com casos de uso seguindo o modelo que ser usado no projeto. Uma especificao determina os requisitos que um usurio tem para o software. Cada funcionalidade do sistema a ser implementado descrita em um caso de uso, ou seja, um cenrio que descreve como essa funcionalidade usada no sistema.

Ao contrrio dos outros laboratrios, voc no deve criar a classe main/menu. Neste laboratrio voc est construindo a base (backend) do sistema. Sua obrigao oferecer classes com mtodos bem definidos para que qualquer desenvolvedor possa pegar o que voc desenvolveu para implementar a interface grfica/textual/web com o usurio (frontend).

Para facilitar a vida do desenvolvedor de frontend, seu sistema deve ter uma classe Facade contendo todos os mtodos e operaes que o desenvolvedor responsvel por implementar a interface grfica ou textual precisa. Exemplo de uma classe Main que poderia ser implementada utilizando a classe Facade que seu sistema ir oferecer:

public class Main {

    public static void main(String[] args) {
        Facade facade = new Facade();

         facade.inicializa();

         facade.adicionaCliente(Joo Neto","SPLAB","joaoneto@ccc.ufcg.edu.br);
        // etc
   }

}

Dica de implementao!

O que Facade/Faade? Facade = Fachada. uma classe que abstrai todo um sistema (ou um conjunto de subsistemas) para facilitar o uso dessas partes abstradas.

A estratgia de criar uma nica classe de entrada (como a classe Facade) bastante comum e facilita a vida do desenvolvedor. Uma classe como a classe Facade pode compor um ou mais controladores do sistema. Exemplo:

public class Facade {

    // ...

    public void salva() {
       pagamentoController.salva();

        cadastroController.salva();
   }

    // ...

}

Idealmente a Facade deve trabalhar apenas com os tipos mais bsicos do sistema. Isso diminui o acoplamento do sistema. Dessa forma, faa mtodos que usem apenas os tipos bsicos nos parmetros e no retorno (Strings, int, double, etc.).

Uma fachada no o mesmo que um controlador. Uma facade deve oferecer apenas mtodos simples e de uso geral do usurio. Um controlador pode ter mais mtodos e responsvel por fazer tratamento da lgica de negcio. Em geral a facade apenas delega aes para um ou mais controladores que o compe.

Sistema para Auto-Gesto de comrcio de Alimentos - SAGA

A comercializao de lanches acontece de maneira auto-gerida em alguns laboratrios de graduao da nossa universidade. Atualmente, vrios fornecedores ofertam os seus produtos alimentcios nos laboratrios, mas no permanecem no local para receber o dinheiro da compra. Na verdade, eles deixam um caderninho, onde cada cliente mantm uma pgina (a chamada *conta*) para anotar as suas compras. Ao retirar um produto, o cliente anota a data, quantidade e o nome do produto na pgina de sua *conta*, no caderninho do fornecedor. Periodicamente, os clientes realizam o pagamento de suas *contas*. Como no h prazo de pagamento estipulado, este pode acontecer ao final da semana, do ms, etc. Infelizmente, alguns fornecedores tem relatado que clientes esto "esquecendo" de pagar as suas contas ou at de anotar a retirada de produtos. Por outro lado, os clientes reclamam a falta de promoes ou outros incentivos ao pagamento de suas contas. Diante disto, surgiu a necessidade de informatizar este processo.

 

A atividade proposta por este Lab implementar um sistema de comrcio eletrnico para dar suporte comercializao de lanches nos laboratrios. As necessidades de fornecedores e clientes motivou o surgimento do SAGA - Sistema para Auto-Gesto de comrcio de Alimentos.

Neste programa, possvel cadastrar os clientes, fornecedores dos lanches e os produtos oferecidos. Alm disso, possvel manter o registro das contas dos clientes junto aos seus fornecedores. Atravs do sistema, possvel totalizar e realizar o pagamento das contas dos clientes. Do ponto de vista do fornecedor, ser possvel fazer consulta sobre os seus recebveis e aplicar promoes nos seus produtos.

Este Lab composto por 3 partes especificadas em formato de Casos de Uso. Nesta parte inicial, especificaremos os casos de uso: (1)CRUD de Clientes; (2)CRUD de Fornecedores e (3)CRUD de Produtos de fornecedores. Estes casos de uso constituem a primeira parte do sistema SAGA. Entretanto, este laboratrio ter mltiplas entregas, em cada uma delas, um conjunto de casos de uso deve ser implementado de acordo com o esquema abaixo:

  • Primeira parte: Sistema completo com os casos de uso 1-3
  • Segunda parte: Sistema completo com os casos de uso 1-6
  • Terceira parte: Sistema completo com os casos de uso 1-7

O que CRUD? um acrnimo que na lngua inglesa que significa: Create Read, Update e Delete. Estas so as operaes bsicas de qualquer sistema de informao que precisa manter e manipular registros das entidades do domnio.

US1. CRUD de Clientes

User story: Como administrador do sistema, quero adicionar, recuperar, editar ou apagar clientes.

O sistema SAGA mantm o cadastro dos clientes dos lanches com informaes suficientes para identific-los e localiz-los na universidade. Os clientes so identificados, unicamente, por seu CPF e possuem outras informaes adicionais como nome, email e localizao (refere-se ao nome do local onde trabalham: LSD, LSI, SPLab, etc). Sempre que for feita uma operao sobre um cliente, este deve ser recuperado atravs do seu CPF. No possvel cadastrar um cliente j existente.

A Facade do sistema deve ter mtodos para:

  • Cadastrar clientes;
  • Retornar a representao textual de um Cliente;

Exemplo:

"Joo Neto - SPLAB - joaoneto@ccc.ufcg.edu.br"

  • Retornar a representao textual de todos os clientes cadastrados no sistema (use o separador | para separar os clientes retornados, em ordem alfabtica do nome do cliente);
    EXEMPLO:
    Ana Silva - Embedded - anasilva@ccc.ufcg.edu.br | Joo Neto - SPLAB - joaoneto@ccc.ufcg.edu.br
  • Editar cadastro de um cliente(Devido a unicidade de CPF, este campo no pode ser editado);
  • Remover cliente do cadastro;

US2. CRUD de Fornecedores

User story: Como administrador do sistema, quero adicionar, recuperar, editar ou apagar fornecedores.

O sistema SAGA tambm mantm o cadastro dos fornecedores dos lanches. Os fornecedores so identificados unicamente pelo seu *nome* e no podem ser cadastrados no sistema em duplicidade. Sempre que for feita uma operao sobre um fornecedor, este deve ser recuperado atravs desta chave. Alm do nome, outras informaes necessrias para o cadastro do fornecedor o seu email e telefone. Cada fornecedor mantm a sua lista dos produtos que comercializa. Informaes mais detalhadas sobre os produtos, sero apresentadas no US3.  

A Facade do sistema deve ter mtodos para:

  • Cadastrar fornecedores;
  • Retornar a representao textual de um Fornecedor;

Exemplo:

"Dona Ins - dines@gmail.com - 83 9999-5050"

  • Retornar a representao textual de todos os fornecedores cadastrados no sistema (use o separador | para separar os fornecedores retornados, em ordem alfabtica do nome do fornecedor);
    EXEMPLO:
    "Dona Ins - dines@gmail.com - 83 9999-5050 | Josenilda - nilda@computacao.ufcg.edu.br - 83 98736-5050 | Ron Weasley - rweasley@splab.ufcg.edu.br - 83 99936-5050"
  • Editar cadastro do fornecedor (Devido a unicidade de nome, este campo no pode ser editado);
  • Remover fornecedor do cadastro;

US3. CRUD dos Produtos dos Fornecedores

User story: Como administrador do sistema, quero adicionar, recuperar, editar ou apagar produtos de fornecedores.

Os produtos comercializados so de responsabilidade dos fornecedores. Assim, todo o controle dos produtos est sempre associado a um fornecedor especfico. O  sistema SAGA permite o cadastro de produtos para os fornecedores j existentes no sistema. Inicialmente, trataremos nesta US de produtos simples. Entretanto, nos prximos casos de uso os produtos podem tornar-se entidades mais complexas: com especializaes e comportamentos distintos.

Produtos devem ter preo, nome e descrio. No deve ser possvel cadastrar o mesmo produto para o mesmo fornecedor. Os produtos podem ser identificados por seu nome e descrio.

A Facade do sistema deve ter mtodos para (em todas as funcionalidades abaixo o fornecedor identificado pelo seu nome e o produto pelo nome e descrio):

  • Cadastrar produto para um fornecedor;
  • Retornar a representao textual de um produto de um fornecedor;

Exemplo <nome, descrio e preo>:

"Tapioca simples - Tapioca com manteiga - R$3,00"

  • Retornar a representao textual de todos os produtos para um dado fornecedor cadastrado no sistema (use o separador | para separar os dados retornados, em ordem alfabtica do nome do produto);
    EXEMPLO:

"Dona Ins - Bolo - Bolo de chocolate - R$3,00 | Dona Ins - Tapioca simples - Tapioca com manteiga - R$3,00 | Dona Ins - Tapioca completa - Tapioca com cco, queijo e manteiga - R$3,50

  • Retornar a representao textual dos produtos de todos os fornecedores cadastrado no sistema (um por linha, em ordem alfabtica do nome do fornecedor);
    EXEMPLO:
    "Dona Ins - Tapioca simples - Tapioca com manteiga - R$3,00 | Dona Ins - Tapioca completa - Tapioca com cco, queijo e manteiga - R$3,50 | Dona Ins - Bolo - Bolo de chocolate - R$3,00 | Josenilda - Mousse - Mousse de Limo - R$4,00 | Josenilda - Salada - Salada de frutas com leite condensado - R$4,50 | Josenilda - Biscoito doce - Maizena - R$3,00"
  • Editar produto (S possvel alterar o preo, para outras alteraes sugere-se excluir o produto e inseri-lo novamente );
  • Remover produto do fornecedor;
